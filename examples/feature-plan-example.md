## `spec.md` — purpose, requirements, decisions, DoD

```markdown
# Spec — Per-team webhook rate limiting

## 1. Purpose

Add per-team request budgets to webhook ingestion so a noisy team can't starve others. Provide an
operator-controlled kill switch to halt ingestion for a specific team during incidents without
deploying.

## 2. User story

- As an **SRE on-call**, I want to flip a kill switch for one misbehaving team, so that I can
  contain an incident without a deploy or a global outage.
- As a **legitimate team**, I want a guaranteed minimum throughput, so that another team's
  spike doesn't starve me.

## 3. Requirements

### Functional
- R1 — Each team has a request budget (default 200 req/min, per-team override config).
- R2 — Denied requests return 429 with `Retry-After` and a structured error body.
- R3 — Audit log entry on every deny: team_id, time, reason, count-in-window, correlation_id.
- R4 — Kill switch (ops-only) disables ingestion for a specific team within 30s, no deploy.

### Non-functional
- NF1 — p99 latency on `/ingest` increases by ≤ 5ms under 500 rps load.
- NF2 — No new high-cardinality metric labels (team_id is bucketed, not raw).

## 4. Goals / Non-goals

**Goals**: per-team isolation; kill switch; audit log.
**Non-goals**: per-endpoint sub-limits; cost-based limiting; new dashboards beyond two new panels.

## 5. Core decisions

- **D1** — Token-bucket algorithm, Redis-backed. ([`decisions.md#d1`](./decisions.md#d1))
- **D2** — Limiter applies **before body parsing** (cheap-reject path). ([`decisions.md#d2`](./decisions.md#d2))
- **D3** — Killswitch defaults to **off** per team; never wide-open. ([`decisions.md#d3`](./decisions.md#d3))
- **D4** — Audit log includes team_id (real), correlation_id, reason, counts — **no payload**.
  ([`decisions.md#d4`](./decisions.md#d4))

## 6. Acceptance criteria

- [ ] Over-limit team gets 429 with `Retry-After`.
- [ ] Killswitch flip halts ingestion for the targeted team within 30s; other teams unaffected.
- [ ] Audit log lines visible in central sink with required fields.
- [ ] p99 latency increase ≤ 5ms under load test.

## 7. Test strategy

Risk-driven: focus on (a) atomicity under concurrent requests against Redis, (b) clock-driven
expiry edges (frozen clock), (c) handler returning correct status + headers, (d) audit log shape.

## 8. Security considerations

- Trust boundary: untrusted webhook payload from the internet.
- Sinks: log line (log-injection if we put raw team-supplied headers in logs), Redis Lua (no
  concat — args parameterized).
- OWASP: A04 (DoS abuse), A05 (misconfig if killswitch wide-opens), A09 (logging — capture
  without leaking PII).
- `security-reviewer` trigger: **end of phase 2** (before flag is turned on in prod).

## 9. Observability plan

Logs (audit), metrics (`webhook.ratelimit.{allow,deny}` and `webhook.killswitch.tripped` —
team_bucket label), traces (span attribute `ratelimit.outcome`), alerts on sustained deny rate.

## 10. Rollout & rollback

Flag `webhook_ratelimit_v1`, default off, 1% → 10% → 50% → 100% with 24h between. Rollback: flag
off. Success metric: zero P0 incidents for 14d; per-team p95 within ±5ms.

## 11. Risks

| Risk                                       | Likelihood | Impact | Mitigation                                     |
|--------------------------------------------|-----------|--------|------------------------------------------------|
| Redis hiccup → false denies                | M         | H      | Fail-open with circuit breaker; alert on spike |
| Limit too tight for legitimate bursts      | M         | M      | 24h ramp + per-team override                   |
| Killswitch misconfig kills all teams       | L         | H      | Per-team only; no "all" key; 2-person approval |

## 12. Definition of Done

- [ ] All phases complete with passing tests.
- [ ] `security-reviewer` consulted at end of phase 2; no Blocker findings.
- [ ] Staging ramp completed (1% → 100%) without incident.
- [ ] Ops runbook updated with kill-switch procedure.
- [ ] Audit log fields documented in `docs/observability/audit-events.md`.
- [ ] `progress.md` status set to `done`.
```

## `decisions.md` — the deliberation behind each choice

```markdown
# Decisions — Per-team webhook rate limiting

## D1 — Rate-limit algorithm + backing store

> Status: `accepted` • Date: 2026-05-20 • Verified with user: yes • Phase: work-item-wide

### Context

The service runs N replicas behind a load balancer; an in-memory limiter cannot enforce a per-team
budget across replicas. We need atomic refill + take semantics.

### Decision

Use a token-bucket algorithm backed by a Redis Lua script (atomic refill + take).

### Alternatives considered

#### Option A — Token bucket in Redis (chosen)
- Pros: smooth bursts, well-understood, ops-friendly, atomic via Lua.
- Cons: a network hop per request (well within latency budget).

#### Option B — In-process leaky bucket (rejected)
- Pros: zero extra deps.
- Cons: drift across replicas, unfair under autoscaling, can't enforce global budget.
- Reason for rejection: violates the goal of *global* per-team isolation across N replicas.

#### Option C — Sliding-window in Redis (rejected, revisit later)
- Pros: more precise (no edge-of-window bursts).
- Cons: more expensive Lua script per request; harder to reason about.
- Reason for rejection: bucket precision is sufficient for the SLA; reconsider if we see bucket
  drift complaints.

### Discovery questions

(User picked A directly; discovery not used.)

### Consequences

- Positive: clear ops story; latency budget preserved with one Redis call per request.
- Negative: Redis hiccup affects every request — circuit breaker required (see D5 below).
- Reversibility: low cost to swap to sliding-window later; same interface, different Lua.

### Implementation notes

- `internal/ratelimit/redis.go` — Lua script lives here.
- Phase 2 wiring.

---

## D2 — Where the limiter applies in the request path

> Status: `accepted` • Date: 2026-05-20 • Verified with user: yes

### Decision

Limiter applies **before body parsing** (cheap-reject path).

### Alternatives considered

#### Option A — Before body parsing (chosen)
- Pros: trivial DoS resistance — we don't allocate a payload buffer for over-limit traffic.
- Cons: can't apply payload-size-aware limits (not in scope per non-goal §4).

#### Option B — After body parsing
- Pros: enables cost-based limiting later.
- Cons: over-limit traffic still allocates and parses; opens trivial resource-exhaustion vector.
- Reason for rejection: cost-based limiting is a non-goal; protecting the cheap path is the
  whole point.

### Consequences

Positive: predictable, cheap rejection path. Negative: forecloses cost-based modes. Reversibility:
moderate — we can move the limiter after parsing if we add cost-based limiting later.

---

## D3 — Killswitch default behavior

> Status: `accepted` • Date: 2026-05-20 • Verified with user: yes

### Decision

Killswitch defaults to **off** for every team; never wide-open by default; no "all teams" key.

### Alternatives considered

#### Option A — Per-team boolean, default off (chosen)
- Pros: safe default; fits OWASP A05 (misconfig); ops can enable for exactly one team.
- Cons: no quick way to halt *all* ingestion in a true emergency (acceptable — we have rate-limit
  fallback for that).

#### Option B — Per-team + "all" override
- Pros: faster total halt during a wide incident.
- Cons: misconfig blast radius is the whole service.
- Reason for rejection: a wide halt is rarer than a single-team halt; we have other levers (deploy
  rollback, LB drain) for the wide case.

(More decisions follow, including D4 for audit log content and D5 for circuit-breaker design.)
```

## What this example demonstrates

- **Six focused files** (plus `phases/` and `reviews/`) instead of one monolithic plan — each
  artifact has a single purpose; readers can jump straight to spec, decisions, progress, or
  diary without scrolling.
- **Decisions verified inline** — each entry in `decisions.md` shows the options considered with
  concrete pros and cons, marks user verification, and (if used) records the discovery Q&A.
- **`spec.md` references decisions by ID** — the spec stays a stable "what + why" document;
  `decisions.md` carries the deliberation.
- **`progress.md` is the front door** — read it first to know where the work item stands; phases
  table tells you what's done, what's next, what's blocked.
- **`diary.md` is the time machine** — when you come back to this work item in three months,
  diary tells you the story of how it evolved.
- **No stale state** — if a constraint discovered during phase 2 forces D1 to change, the agent
  supersedes D1 with D5 in `decisions.md`, updates the one-liner in `spec.md` §5, updates
  `phases/02-…md` if it shifts the shape, and appends a `diary.md` entry recording why.
